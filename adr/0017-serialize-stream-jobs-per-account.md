# ADR-0017: 同一アカウントの配信系ジョブは直列実行に制限する

## Status
Accepted

## Context
Issue #16 は、同じアカウントで複数の配信・中継セッションを同時に開くと、後から入ったセッションが先行セッションを停止させる挙動を、既知の仕様または既知バグとして扱い、設計制約へ反映すべきだと指摘している。

この制約は、配信開始フローの細部より上位にある運用安全性の問題である。ADR-0012 が CLI を正式制御面にし、ADR-0016 がログイン前分岐を整理しても、同じアカウントへ並列にジョブを投げられる限り、正常系ジョブが後続ジョブに破壊される危険は残る。

現時点では、後勝ちセッション衝突をアプリ側で明確に識別できる保証もない。自動リトライやAIエージェントの並列検証まで含めると、事故原因の切り分けが難しくなるため、まずは安全側へ倒す必要がある。

## Decision
初期PoCおよび基盤設計では、同一アカウントの配信系ジョブを直列実行に制限する。

具体方針は次とする。

- アカウントID単位の排他ロックを前提にする。
- ロック対象は、配信開始、ハンカチ中継開始、配信設定画面遷移、再試行、復旧操作を含む「配信系ジョブ」全体とする。
- 既存ロックがある場合、後続ジョブは既定で失敗させる。
- 明示 override は将来候補とし、初期段階では採用しない。
- 自動リトライは、新規セッションを起こす前に既存セッション状態を確認できるまで無効化または保守的に制限する。
- CI、AIエージェント、複数端末で同じアカウントを共有しないことを運用前提として文書化する。

ロック保持場所は未確定でもよいが、少なくとも CLI / backend / job queue のどこか1層で「同一アカウントは同時に1ジョブのみ」を必ず保証する。

## Consequences
既存配信を誤って停止させる事故を大きく減らせる。障害時も、別セッション衝突を設計上まず疑うべき状況が減るため、ログ解釈がしやすくなる。

一方で、同一アカウントを使った並列検証や高速リトライはできなくなる。複数並列が必要なら、別アカウント運用やアカウントプール設計が必要になる。

## Alternatives Considered

| Option | Pros | Cons | Current Judgment |
| --- | --- | --- | --- |
| 同一アカウントを直列実行に制限する | 最も安全。設計が明快 | 並列性が落ちる | 採用 |
| 後続ジョブで既存配信を上書きできるようにする | 運用は柔軟 | 既存配信破壊リスクが高い | 初期段階では不採用 |
| セッション衝突検知ができるまで並列実行を許す | 実験自由度は高い | 原因切り分け不能。事故率が高い | 不採用 |

## Risk Notes
- ロックTTLやクラッシュ回収が曖昧だと、今度はゾンビロックで停止する。
- アカウント識別子の定義がぶれると、別表記で同一アカウントを並列実行してしまう。
- 運用文書に反映しないと、CIや人手オペレーションで簡単に破られる。

## Validation Plan

### Phase A: Lock Semantics

1. ロック対象ジョブ一覧を作る。
2. 取得、更新、解放、TTL、手動解除のルールを定義する。
3. 実行中ジョブIDとアカウントIDをログへ残す。

Success criteria:

- 同一アカウントの並列実行可否を機械的に判定できる。
- ロックが残留した場合の回収手順がある。

### Phase B: Failure Classification

1. 正常終了、クラッシュ、通信断、別セッション衝突疑いのケースを分けて記録する。
2. 自動リトライが既存配信を止めない条件を整理する。

Success criteria:

- リトライ前にロックとセッション状態を確認する設計がある。
- 停止原因として「別セッション衝突疑い」をログ上で追える。

## Next Action
次は、アカウント単位ロックの保持層とTTL方針をPoC仕様へ追加し、CI/AIエージェント向け運用注意として明文化すること。
