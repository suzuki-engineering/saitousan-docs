# ADR-0015: 配信開始導線の最小PoCはホーム画面から中継開始設定画面到達までに限定する

## Status
Proposed

## Context
Issue #14 は、ログイン済みホーム画面から「ハンカチ中継」経由で中継開始設定画面まで進む操作を、Appiumで安定して扱えるようにしたいというものだ。

既存ADRでは、ADR-0008 がOTPログインを人手ゲートとし、ADR-0010 と ADR-0014 が配信開始の契約面を整理している。しかし、配信設定値をどれだけ整理しても、ホーム画面から正しい導線で設定画面へ入れなければ実装は前に進まない。

一方、この段階でタイトル入力や通知設定まで一気に扱うと、ホーム画面の要素特定、画面遷移判定、設定入力、配信開始確認が1つの失敗面に混ざる。初期PoCでは、まず「設定画面まで到達できるか」を独立した境界として確定する必要がある。

## Decision
Appiumの最小PoCでは、ホーム画面から中継開始設定画面まで到達する導線を独立フローとして先に成立させる。

初期フローの責務は次までに限定する。

- 前提は「ログイン済みでホーム画面にいる」こと。
- ホーム画面で「ハンカチ中継」を誤タップなく特定する。
- ハンカチ中継画面へ遷移したことを判定する。
- 画面下部の「中継する」を押す。
- 中継開始設定画面へ到達したことを完了とする。

次はこのフローに含めない。

- OTPログインや再ログイン処理
- 配信設定項目の入力
- 実際の配信開始確定
- 配信失敗時の復旧や再試行

## Consequences
設定入力以前の画面遷移が独立して検証できるため、ホーム画面のselector問題と設定項目の契約問題を分離できる。PoCの失敗時も、「設定画面へ入れない」のか「設定内容を反映できない」のかを切り分けやすくなる。

一方で、設定画面到達だけでは実運用価値はまだ限定的である。下部ボタン位置、スクロール有無、画面文言の変化など、実機差分の影響は残る。

## Alternatives Considered

| Option | Pros | Cons | Current Judgment |
| --- | --- | --- | --- |
| 設定画面到達までを独立PoCにする | 失敗面を分離しやすい | 単体では価値が限定的 | 採用 |
| 設定入力と配信開始まで一気に扱う | 完成形に近い | 切り分け困難。失敗面が多い | 初期段階では不採用 |
| ホーム画面探索を手動前提にする | 実装が軽い | 自動化の価値が薄い | 不採用 |

## Risk Notes
- ホーム画面の導線名やレイアウト変更で selector が壊れやすい。
- 「中継する」ボタンが fold 下にある場合、スクロール戦略が必要になる。
- 未ログイン時に同じ操作を実行すると、誤って別画面へ進む恐れがある。

## Validation Plan

### Phase A: View-State Definition

1. ホーム画面、ハンカチ中継画面、設定画面の識別要素を定義する。
2. 各画面で期待するアクセシビリティ文言、ボタン名、補助表示を記録する。

Success criteria:

- 各画面の完了判定が1つ以上ある。
- selectorが単一要素に依存しすぎていない。

### Phase B: Repeated Navigation Check

1. 同一端末でホーム画面から設定画面まで複数回繰り返す。
2. 誤タップ、要素未発見、スクロール要否を記録する。
3. 失敗時にどの画面で止まったかを artifact に残す。

Success criteria:

- 複数回の遷移で大きくぶれない。
- 失敗時に停止地点をログとスクリーンショットで説明できる。

## Next Action
次は、ホーム画面と中継画面の view-state 判定条件を `research/` またはPoCメモに書き出し、最低1回の往路遷移を実機で確認すること。
