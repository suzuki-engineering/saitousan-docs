# ADR-0008: OTPログインは初期段階では人手ゲートとして扱う

## Status
Accepted

## Context
Issue #8 は、斉藤さんアプリのログインがメールOTP前提であり、CLIや自動化からどう扱うべきかを問うている。

OTPログインは、アプリ内操作だけでは完結しない。メール送信、受信確認、コード入力という外部依存を含むため、単一の非対話ジョブとして隠蔽しようとすると、認証情報管理、メールアクセス権、期限切れ、誤入力、再送といった失敗要因を一度に抱え込む。

このリポジトリの現段階では、Android実行環境と配信経路の成立確認が優先であり、メールボックス自動読取まで同時に進める合理性は低い。

## Decision
初期PoCでは、OTP取得は人手ゲートとして扱う。

ログイン制御は次の2段階に分離して記録する。

- `login start`: メールアドレス入力と確認コード送信
- `login complete`: OTP入力とログイン完了確認

一括コマンドでの完全自動化は初期段階では採用しない。CLIまたは将来のAPIは、OTP待ち状態を明示的に返し、人手介入後に再開できる形を前提にする。

自動OTP取得は将来候補として残すが、条件は次の通りとする。

- 専用メールボックスを使う
- 読取権限を最小化する
- 認証情報をコードやリポジトリに保存しない
- 誤取得時に人手で停止できる

## Consequences
初期PoCの複雑さを抑えつつ、ログイン失敗原因を分離しやすくなる。OTP関連障害を、Android UI障害や配信障害と切り離して評価できる。

一方で、完全無人実行はできない。CIやスケジュール実行へ進むには、別途メール取得経路の設計と安全確認が必要になる。

## Alternatives Considered

| Option | Pros | Cons | Current Judgment |
| --- | --- | --- | --- |
| OTPを人手入力にする | 最も安全で分かりやすい | 無人実行できない | 採用 |
| 単一コマンドで入力待ちする | 操作は単純に見える | 状態遷移と再開が曖昧になる | 初期段階では不採用 |
| メールボックスから自動取得する | 無人化に近づく | 認証情報管理と誤取得リスクが増える | 将来候補 |

## Risk Notes
- OTPの期限切れや誤入力で、再送や再実行の導線が必要になる。
- ログイン済み状態の判定が曖昧だと、不要なOTP送信が増える。
- 自動取得へ進む際は、個人メール利用や広いOAuth権限がリスクになる。

## Validation Plan

### Phase A: Manual OTP Flow

1. `login start` でコード送信まで進める。
2. 人手でメールを確認する。
3. `login complete` でOTPを投入する。
4. 成否、再送、期限切れ時の分岐を記録する。

Success criteria:

- 2段階フローでログイン完了まで到達できる。
- OTP待ち状態をCLI/APIの出力で識別できる。

### Phase B: Session Reuse Check

1. 再起動後もログイン状態が維持されるか確認する。
2. どの条件で再ログインが必要になるか整理する。

Success criteria:

- 再ログイン条件が文書化される。
- 配信開始前の事前チェック項目に組み込める。

## Next Action
次の作業は、`login start` / `login complete` / `status` の状態遷移をPoC仕様へ落とし、OTP待ち・期限切れ・再送のログ記録項目を定義すること。
